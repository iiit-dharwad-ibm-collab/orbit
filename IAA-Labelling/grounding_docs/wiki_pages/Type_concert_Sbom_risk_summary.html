Workflow: Assessing package risk and reliability

Source: concert-2.2.x-documentation.md (concert) The Software composition dimension helps you understand the quality, reliability, and risk associated with your software packages and their associated licenses. After generating and uploading a package SBOM file in CycloneDX format, Concert assesses the risk and calculates a aggregate reliability score to help you prioritize action. The following steps outline the workflow for analyzing package SBOM data through the Software composition dimension. Step 1: Generate a package SBOM Refer to Generating a package SBOM (CycloneDX) for instructions on generating a package SBOM in CycloneDX format.

Generating a package SBOM (CycloneDX)

Source: concert-2.2.x-documentation.md (concert) Generate and upload a package SBOM file in CycloneDX format to assess the reliability, maintenance, licensing, and security of the software package components used in your applications. After you upload packages, you can view them as Correlated or Uncorrelated components.

Application SBOM (ConcertDef schema)

Source: concert-2.2.x-documentation.md (concert) The application SBOM focuses on the static definition of the application's components, libraries, and dependencies. It provides a high-level, theoretical list of these components and their metadata using the ConcertDef schema.

Refreshing a package SBOM (CycloneDX)

Source: concert-2.2.x-documentation.md (concert) The SBOM package that you upload has details such as OpenSSF scorecard, Recommendations, File view, History, and others. But the score is generated at the time that the package is added and is not dynamically refreshed. Now, you can refresh your package details on demand with Concert.
